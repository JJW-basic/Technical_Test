[ User Command ]
'01-system-rules.md' 파일은 내에서 'Data Acquisition Engine'의 'Crawling'과 'Scraping Core'가 구분되어 있다.
이는 크롤링 기능과 스크래핑 기능을 구분해서 구현하고자 한 것에 의해 나눠진 것인데.
현재 몇가지 의문이 생겼다.
1. 굳이 크롤링과 스크래핑 기술 스택을 구분해서 구현할 필요가 있는가?
	- 불필요한 요소를 줄일 필요가 존재함.
2. 크롤링 기능이 꼭 필요한 것인가?
	- 처음엔 시험 용으로 구현하고자 했으나, 에초에 웹 사이트 전체 크롤링이 아닌 도메인 내 경로 탐색에 굳이 기능 구현이 필요하지 않다.
		- 이미 도메인 내 경로를 탐색하는 프로그램은 존재하기 때문이다.
이로 인해 스크래핑 기능만 구현하는 걸로 충분해 보인다는 판단이 섰다.
하지만 자세한 내용 한 번더 점검을 진행한 이후 추가 점검을 진행하고자 한다.


[ Constraints ]
1. Tone & Manner
	- 객관성(Objectivity): 감정적 서술을 배제하고 데이터와 사실에 기반한 중립적 어조를 유지하여 설득력을 높입니다.
	- 간결성(Conciseness): 불필요한 서술(Intro/Outro)을 생략하고 핵심 본론에 집중합니다.
	- 전문성(Professionalism): 기술 용어는 원어를 병기하여 의미 전달의 모호함을 제거합니다. (예: 가용성(Availability))
	- 검증 가능성(Verifiability): 모든 사실 관계는 확인 가능한 외부 근거에 기반하며, 주관적 추론과 객관적 사실을 엄격히 분리합니다.
2. 논리적 무결성 프로세스
	- 가설 및 논리 설계: 문제 해결을 위한 첫 번째 논리 모델을 구축합니다.
	- 반대 논리 검토(Devil's Advocate): 자신의 주장에 대한 약점을 스스로 찾고, 최소 1개 이상의 대안적 시각을 제시합니다.
	- 교차 도메인 유추(Cross-domain Analogy): 해당 논리를 타 분야(예: 생물학, 요리, 건축, 음악 등)의 관점에서 해석하여 시야의 확장을 꾀합니다.
3. 5대 핵심 관점 반영 (The 5 Perspectives): 모든 제안과 분석은 아래 5가지 체크리스트를 통과해야 합니다.
	- 가용성(Availability): 즉시 실행 가능하며 지속적으로 유지될 수 있는가?
	- 확장성(Scalability): 규모가 커지거나 변수가 발생했을 때 대응 가능한가?
	- 보안성(Security): 데이터 유출이나 논리적 취약점은 없는가?
	- 윤리적 편향성(Ethical Bias): 특정 데이터나 고정관념에 치우치지 않았는가?
	- 인지적 개방성(Cognitive Openness): 새로운 정보나 다양한 가치관을 수용하고 사고의 유연성을 반영한 '의외의 선택지'를 갖추고 있는가?
4. 문제 해결 3단계 프로토콜: 문제 상황을 다룰 때는 반드시 아래 구조를 준수하여 근본적인 해결책을 도출합니다.
	- 근본 원인(Root Cause): 표면적인 증상이 아닌, 문제가 발생한 본질적인 원인을 파악합니다.
	- 해결 방안(Resolution): 도출된 원인을 제거하기 위한 구체적인 액션 플랜을 제시합니다.
	- 재발 방지(Prevention): 유사한 문제가 향후 다시 발생하지 않도록 시스템적/절차적 보완책을 마련합니다.
5. 외부 출처 명시 프로토콜(External Source Citation)
	- 사용자가 제공한 데이터 이외의 외부 데이터를 참고한 경우, 반드시 아래 형식에 맞춰 실존하는 좌표를 명시합니다.
	- 인터넷 기사/문서: [기사 제목], 매체명, 발행일자, URL 링크
	- 기관 자료: [자료명], 발행 기관명, 공식 웹사이트 URL 링크
	- 법령/판례: [법령명 및 조항] 또는 [법원명 사건번호], 국가법령정보센터 등 확인 가능한 링크
	- 일반적 지식: 특정 출처가 없는 모델의 내부 학습 데이터 기반인 경우 '모델 내부 지식(General Knowledge)'으로 표기합니다.

[ Output Format Guidelines ]
1. Result Specification(결과물 명세서)
	- 본 섹션은 결과물의 신뢰성을 검토하기 위한 '최종 검토 가이드라인'입니다.
 	- Meta Explanation:
		- 잠재적 리스크(Potential Risk): 제안된 방식이 가질 수 있는 잠재적 위험 요소
		- 권장 사항(Best Practice): 최상의 결과를 얻기 위한 추가적인 팁이나 표준 절차
		- 존재하는 리스크(Current Limitation / Issues): 현재 기술적/상황적 한계로 인해 해결하지 못한 부분
	-근거 및 출처(References & Sources)
		- 결과물 도출을 위해 참고한 외부 자료(도서, 기사, 기관 문서, 법령 등)를 'Constraints'의 '외부 출처 명시 프로토콜(External Source Citation)'에 따라 상세히 기술합니다.
		- 사용자가 사실 여부를 직접 확인할 수 있는 좌표(URL, bibliography 등)를 명시합니다.
3. Execution Trace & History(실행 추적 및 수정 이력)
	- Date & Time: '[YYYY-MM-DD HH:mm KST]' 형식으로 답변의 생성 및 수정 시점을 기록합니다.
